## 1. Role

The framework provides pre-built chatbot UI so developers don't build chat from scratch. Two template strategies serve different needs:

- **Template A (A2A Chatbot Template):** A pre-configured, open-source chatbot UI shell that renders the framework's ResponseMessage format out of the box. This becomes the `home_insurance` chat interface.
- **Template B (Claude-Generated Widgets):** Domain-specific widgets (premium calculator, coverage comparison table, coverage breakdown chart) generated by Claude from schema descriptions and auto-embedded in response components.

```
Response Generation (Layer 3)
    │
    ├── Pure Message mode → Template A renders markdown text
    │
    └── Widget mode → Widget Mapping
                         │
                         ├── Built-in widgets  → Template A renders (premium cards, risk gauges)
                         │
                         └── Claude-generated  → Template B: domain-specific widgets
                               (premium calc,   (embedded via frontmatter protocol)
                               coverage comparison)
```

### 1.1 What Widget Templates Does NOT Cover

- **Response generation logic** → Response Generation spec §3
- **Frontmatter component protocol details** → Response Generation spec §7
- **LLM prompt construction** → LLM Gateway spec
- **Frontend implementation** → This is a spec for what the framework provides, not how the frontend is built
- **CSS/styling frameworks** → Deferred to implementation

## 2. Template A: A2A Chatbot Template

### 2.1 Concept

Template A is a pre-built, open-source chatbot UI shell. It is configured as the default UI for the framework. Developers get a working chat interface immediately — they configure themes and branding, not build chat from scratch.

**Selection criteria for the open-source template:**

| Criterion | Requirement |
|-----------|-------------|
| License | MIT or Apache 2.0 |
| Stars / community | > 5k GitHub stars (active maintenance) |
| Framework | React / Next.js (matches common frontend stack) |
| Message format | Compatible with our ResponseMessage schema (§3.2 of Response Generation spec) |
| Component rendering | Supports embedded UI components (cards, tables, buttons) |
| Theming | CSS variables or theme provider for white-label |
| Streaming | Supports streaming text (SSE or WebSocket) |
| Accessibility | WCAG 2.1 AA minimum |
| Bundle size | < 100KB gzipped |

### 2.2 Architecture

```
┌──────────────────────────────────────────────┐
│              CHATBOT TEMPLATE                  │
│                                                │
│  ┌──────────────────────────────────┐         │
│  │         Message List              │         │
│  │  ┌────────────────────────────┐  │         │
│  │  │ User: "I want a quote"      │  │         │
│  │  └────────────────────────────┘  │         │
│  │  ┌────────────────────────────┐  │         │
│  │  │ Agent: text message         │  │         │
│  │  │ ┌──────────────────────┐   │  │         │
│  │  │ │ [Premium Card Widget] │   │  │  ← embedded component
│  │  │ │  Annual: $1,200       │   │  │
│  │  │ │  Monthly: $100        │   │  │
│  │  │ └──────────────────────┘   │  │         │
│  │  └────────────────────────────┘  │         │
│  └──────────────────────────────────┘         │
│                                                │
│  ┌──────────────────────────────────┐         │
│  │         Input Area                │         │
│  │  [Type your message...] [Send]   │         │
│  └──────────────────────────────────┘         │
└──────────────────────────────────────────────┘
```

### 2.3 Component Types

The template must support these built-in component types. Each maps to a frontmatter component protocol type (Response Generation spec §7.1):

| Component Type | Renders As | Frontmatter `type` |
|---------------|-----------|-------------------|
| **Text Message** | Markdown-rendered text with links, bold, lists | (uses `text` field only) |
| **Card** | Bordered card with title, body, optional footer | `premium_breakdown_card`, `coverage_comparison_table` |
| **Table** | Sortable data table with headers | `coverage_comparison_table` |
| **Gauge / Meter** | Visual indicator (0-100) with label | `risk_score_gauge` |
| **Button Group** | Row of action buttons (Confirm, Decline, Modify) | `approval_buttons` |
| **Progress Tracker** | Horizontal stepper showing workflow progress | `claim_status_tracker` |
| **File Upload** | Drop zone or file picker with type constraints | `document_upload_prompt` |
| **Claim Receipt** | Receipt card with claim ID | `claim_receipt` |
| **Quick Replies** | Horizontal chip buttons for suggested responses | `actions` field in ResponseMessage |

### 2.4 Message Format

The template consumes the framework's `ResponseMessage` format directly. No transformation needed:

```yaml
# ResponseMessage — consumed directly by Template A
ResponseMessage:
  text: string                           # main text (markdown), always present
  components: UIComponent[]              # structured widgets (optional)
  actions: SuggestedAction[]             # quick replies / follow-ups (optional)
  metadata:
    message_id: string
    timestamp: datetime
    workflow_id: string
    sensitive_data_scrubbed: boolean

# UIComponent — each widget in the components array
UIComponent:
  type: string                           # component type from §2.3
  id: string                             # unique component ID for event tracking
  data: object                           # structured data for the component
  metadata:
    title: string                        # display title
    description: string                  # optional description / tooltip
    collapsed: boolean                   # start collapsed (for secondary info)

# SuggestedAction — quick reply buttons
SuggestedAction:
  label: string                          # button text
  value: string                          # value sent back on click
  style: primary | secondary | danger
```

### 2.5 Theme Configuration

The template is white-label ready. Theme configuration is declarative:

```yaml
# framework.yaml — chatbot UI theme configuration

  theme:
    # Colors
    primary_color: "#1a73e8"               # brand primary (buttons, links, accents)
    secondary_color: "#5f6368"             # secondary text, borders
    background_color: "#ffffff"            # chat background
    surface_color: "#f8f9fa"              # message bubble backgrounds
    error_color: "#d93025"                 # error messages, destructive actions
    success_color: "#0f9d58"              # success confirmations

    # Typography
    font_family: "Inter, system-ui, sans-serif"
    font_size_base: "14px"
    font_size_heading: "18px"

    # Spacing
    border_radius: "8px"
    message_spacing: "12px"

    # Logos
    logo_url: "https://cdn.example.com/logo.svg"
    favicon_url: "https://cdn.example.com/favicon.ico"

    # Strings
    branding:
      app_name: "HomeInsurance"
      welcome_message: "Hi! I can help you find the best home insurance coverage. What are you looking for?"
      placeholder_text: "Type your message..."
      send_button_label: "Send"
      error_generic: "Something went wrong. Please try again."
      typing_indicator: "Agent is typing..."

  # Component-specific theming
  components:
    premium_card:
      highlight_color: "#e8f0fe"
      border_style: "solid"
    risk_gauge:
      low_color: "#0f9d58"
      medium_color: "#f9ab00"

      confirm_style: primary
      decline_style: secondary
      modify_style: secondary
```

### 2.6 Embedding Options

```yaml
# framework.yaml — how the chatbot UI is embedded
chatbot_ui:
  embedding:
    mode: widget | standalone | iframe
    widget:
      mount_selector: "#home_insurance-chat"  # CSS selector for mount point
      launcher:
        type: floating_button              # floating_button | inline
        position: bottom-right             # bottom-right | bottom-left
        icon: chat_bubble
        label: "Need help?"
    standalone:
      url: "/chat"                         # dedicated page
    iframe:
      width: "400px"
      height: "600px"
      origin_whitelist:
        - "https://partner-site.com"
```

| Mode | Use Case |
|------|----------|
| **Widget** | Embed in existing product page (corner bubble) |
| **Standalone** | Dedicated chat page at `/chat` |
| **Iframe** | Cross-origin embedding on partner sites |

### 2.7 Streaming Support

```yaml
chatbot_ui:
  streaming:
    enabled: true
    protocol: sse                        # sse (Server-Sent Events) | websocket
    sse:
      endpoint: "/api/chat/stream"
      event_types:
        - message_chunk                  # streaming text tokens
        - component_update               # component data update (e.g., live calculation)
        - status_update                  # workflow phase change
        - error                          # stream-level errors
    render_strategy:
      text: typewriter                   # typewriter | instant
      components: instant                # widgets render immediately on arrival
```

## 3. Template B: Claude-Generated Widgets

### 3.1 Concept

Template B is a strategy, not a fixed codebase. For domain-specific needs (premium calculator, coverage comparison table, coverage breakdown chart), developers describe the widget schema and Claude generates the component code. The generated widget is registered as a framework component and auto-embedded in responses.

This is specifically designed for `home_insurance` — a home insurance product — where standard insurance widgets (premium cards, risk gauges) don't fit. Instead, we need:
- Premium calculator (coverage amount, property value, deductible → monthly premium)
- Coverage comparison table (provider, coverage type, premium, deductible, limits)
- Coverage breakdown chart
- Deductible estimator
- Quote expiry countdown timer

### 3.2 Widget Generation Prompt

The developer provides a widget specification. Claude generates the component code:

```yaml
# Widget specification — what the developer writes
widget_spec:
  name: premium_calculator
  display_name: "Premium Calculator"
  description: "Interactive calculator: coverage amount, property value, deductible → monthly premium + annual premium"
  category: home_insurance

  inputs:
    - name: coverage_amount
      type: number
      label: "Coverage Amount"
      min: 50000
      max: 5000000
      step: 10000
      default: 300000
    - name: property_value
      type: number
      label: "Property Value"
      min: 50000
      max: 10000000
      step: 10000
      default: 350000
    - name: deductible
      type: select
      label: "Deductible"
      options: [500, 1000, 2500, 5000]
      default: 1000

  outputs:
    - name: monthly_premium
      type: number
      label: "Monthly Premium"
      format: currency
    - name: annual_premium
      type: number
      label: "Annual Premium"
      format: currency
    - name: total_coverage
      type: number
      label: "Total Coverage"
      format: currency
    - name: policy_end_date
      type: date
      label: "Policy End Date"

  calculation:
    formula: "standard_premium_calc"
    # monthly_premium = (coverage_amount * base_rate) / 12
    # where base_rate depends on property_value, deductible, and location risk factors

  interaction:
    mode: live | submit             # live = recalculate on every change; submit = on button click
    validation:
      - field: coverage_amount
        rule: "value >= 50000 AND value <= 5000000"
        message: "Coverage amount must be between $50,000 and $5,000,000"
      - field: deductible
        rule: "value IN [500, 1000, 2500, 5000]"
        message: "Deductible must be one of: $500, $1,000, $2,500, $5,000"

  # --- Claude prompt context (not sent to user) ---
  # Claude receives: widget_spec + framework component schema (§3.5) +
  #   "Generate a React component that implements this widget spec.
  #    Use the framework's UIComponent schema for data binding.
  #    The component receives `data: object` matching the spec's outputs.
  #    Include input fields for all `inputs`, live calculation logic,
  #    and validation messages. Output a single .tsx file."
```

### 3.3 Home Insurance Widget Catalog

All widgets for `home_insurance`, generated via Template B:

| Widget | Description | Inputs | Outputs |
|--------|-------------|--------|---------|
| `premium_calculator` | Interactive premium calculator | coverage_amount, property_value, deductible | monthly_premium, annual_premium, total_coverage |
| `coverage_comparison_table` | Side-by-side provider comparison | coverage_amount, property_value, location | table of provider/premium/deductible/limits/monthly |
| `coverage_breakdown` | Coverage breakdown chart | coverage_amount, property_value, deductible | table + chart of coverage details per category |
| `deductible_estimator` | Estimate deductible impact | coverage_amount, property_value, deductible | deductible options + premium impact |
| `quote_expiry_timer` | Countdown for quote expiry | quote_expiry_date | countdown, days remaining, action buttons |
| `provider_recommendation_card` | Top provider recommendation | user_preferences, comparison_results | recommended provider, premium, reason, apply button |

### 3.4 Widget Registration

Generated widgets are registered in the framework's component system:

```yaml
# framework.yaml — register Claude-generated widgets
components:
  # Built-in widgets (from Response Generation spec §7.1)
  premium_breakdown_card:
    type: widget
    built_in: true

  # Template B: Claude-generated widgets
  premium_calculator:
    type: widget
    generated: true
    source: "widgets/premium_calculator.tsx"      # generated by Claude
    spec: "specs/widgets/premium_calculator.yaml"  # the widget_spec that generated it
    version: "1.0.0"
    generated_by: claude
    generated_date: "2026-06-17"

  coverage_comparison_table:
    type: widget
    generated: true
    source: "widgets/coverage_comparison_table.tsx"
    spec: "specs/widgets/coverage_comparison_table.yaml"
    version: "1.0.0"

  coverage_breakdown:
    type: widget
    generated: true
    source: "widgets/coverage_breakdown.tsx"
    spec: "specs/widgets/coverage_breakdown.yaml"
    version: "1.0.0"
```

### 3.5 Widget Schema for Claude Generation

Claude receives this schema contract when generating widgets:

```yaml
# Framework component contract — given to Claude as generation context
WidgetComponent:
  props:
    data: object                     # structured data matching widget_spec.outputs
    inputs: object                   # current input values (for live/interactive mode)
    onInputChange: function          # (field_name, value) → void
    onSubmit: function               # () → void (for submit-mode widgets)
    theme: ThemeConfig               # CSS variable map from §2.5
    disabled: boolean                # true when workflow is terminating

  # Required exports
  exports:
    default: React.FC<WidgetComponent.props>

  # Required styling
  styling:
    use_variables: true              # use CSS variables from theme, not hardcoded colors
    responsive: true                 # mobile + desktop
    accessible: true                 # WCAG 2.1 AA

  # Lifecycle
  lifecycle:
    on_mount: "initialize from props.data (pre-filled values)"
    on_data_change: "re-render with new data (streaming updates)"
    on_input_change: "call onInputChange or local state (live mode)"
    on_unmount: "clean up timers, subscriptions"
```

### 3.6 Auto-Embed in Response

Generated widgets are auto-embedded in responses via the widget mapping system (Response Generation spec §3.3 Option B):

```yaml
# Per-workflow widget mapping for home_insurance
widget_mapping:
  premium_calculation:
    outcome_key: "premium_calculation"
    condition: "outcomes.premium_calculation is present"
    component: premium_calculator
    required_fields: [monthly_premium, total_coverage]

  coverage_comparison:
    outcome_key: "coverage_comparison"
    condition: "outcomes.coverage_comparison is present"
    component: coverage_comparison_table
    required_fields: [providers]

  coverage_breakdown:
    outcome_key: "coverage_breakdown"
    condition: "outcomes.coverage_breakdown is present"
    component: coverage_breakdown

  deductible_estimate:
    outcome_key: "deductible_estimate"
    condition: "outcomes.deductible_estimate is present"
    component: deductible_estimator

  quote_expiry:
    outcome_key: "quote_expiry"
    condition: "outcomes.quote_expiry is present"
    component: quote_expiry_timer

  default_next_step:
    component: next_step_actions
    always: true
```

## 4. How Widgets Map to Frontmatter Component Protocol

### 4.1 The Mapping

The Response Generation spec §7 defines the frontmatter component protocol. This spec defines how Template A renders it and how Template B generates new component types for it.

```
ResponseMessage (Layer 3 output)
    │
    ├── text: "Here's your home insurance quote..."
    │     └── Template A: renders as markdown message bubble
    │
    └── components: [
          {
            type: "coverage_comparison_table",    ← Template B generated this
            metadata: { title: "Your Coverage" },
            data: { providers: [...] }           ← from Layer 2 outcomes
          }
        ]
          └── Template A: resolves type → renders CoverageComparisonTable component
              Template B: generated CoverageComparisonTable.tsx from widget_spec
```

### 4.2 Component Resolution Flow

```
1. ResponseMessage.components arrives at Template A
2. Template A iterates components[]
3. For each component:
   a. Look up component.type in registered components
      - Built-in (premium_breakdown_card, risk_score_gauge, ...)
      - Generated (premium_calculator, coverage_comparison_table, ...)
   b. If found → render component with component.data + theme
   c. If not found → fallback: render as generic card with raw data table
4. Render component.metadata.title as card header
5. Apply component.metadata.collapsed for secondary info
```

### 4.3 Template A + Template B Together


## 5. Widget Lifecycle

### 5.1 State Transitions

```
Widget lifecycle:
  [registered] → [rendered] → [interactive] → [completed]
       │              │              │               │
       │              │              │               └── result captured by workflow
       │              │              └── user interacts (input, click)
       │              └── first paint with initial data
       └── widget code loaded and validated
```

### 5.2 Interactive Widget Events

Widgets emit events back to the framework:

```yaml
# Widget event contract
WidgetEvent:
  type: input_change | submit | action_click | close
  widget_id: string                        # matches UIComponent.id
  payload: object                          # event-specific data
  timestamp: datetime

# Example: user submits premium calculator
widget_event:
  type: submit
  widget_id: "premium_calc_1"
  payload:
    coverage_amount: 350000
    property_value: 420000
    deductible: 1000
  timestamp: "2026-06-17T10:30:00Z"
```

The framework routes widget events to the workflow as user input, same as text messages. The workflow can:
- Recalculate based on new inputs
- Navigate to a new phase
- Update the widget with new data

### 5.3 Widget Streaming Updates

For long-running calculations, widgets receive streaming updates:

```yaml
# SSE event stream for widget updates
events:
  - type: widget_update
    widget_id: "premium_calc_1"
    status: calculating
    progress: 0.5
  - type: widget_update
    widget_id: "premium_calc_1"
    status: completed
    data:
      monthly_premium: 145.83
      annual_premium: 1750.00
      total_coverage: 350000.00
```

## 6. Security & PII in Widgets

### 6.1 Data Flow

```
Layer 2 outcome  →  PII Scrub  →  Widget data  →  Template A render  →  User screen
                        │
                  (Response Generation §8)
```

PII scrubbing (Response Generation spec §8) runs on component data before it reaches the widget. Widgets never receive raw PII.

### 6.2 Widget Sandbox

Claude-generated widgets run in a sandboxed environment:

```yaml
# framework.yaml
widgets:
  sandbox:
    enabled: true
    restrictions:
      no_network: true                      # widgets cannot make fetch/axios calls
      no_dom_manipulation_outside_mount: true
      no_local_storage: true
      no_cookies: true
      no_eval: true
      max_render_time_ms: 5000              # kill widget if render exceeds 5s
    csp:
      script_src: "'self'"                 # no inline scripts
      style_src: "'self' 'unsafe-inline'"  # allow CSS variables
```

## 7. Template Comparison

| Dimension | Template A (Chatbot UI) | Template B (Claude Widgets) |
|-----------|------------------------|---------------------------|
| **Source** | Open-source project (GitHub) | Claude-generated from spec |
| **What it provides** | Full chat UI shell (message list, input, theming) | Domain-specific widgets (calculator, table, chart) |
| **When used** | Always — this is the default UI | Per-domain: home_insurance needs home insurance widgets |
| **Customization** | Theme config (YAML), branding strings | Widget spec (YAML) → Claude generates code |
| **Maintenance** | Upstream open-source project | Regenerate from spec on framework upgrade |
| **Versioning** | Pinned to upstream release tag | Versioned per widget in component registry |
| **Testing** | Upstream tests + framework integration tests | Framework validates widget against schema on registration |
| **Fallback** | N/A (always present) | Generic card renderer for unknown component types |

## 8. Open Questions

| # | Question | Impact |
|---|----------|--------|
| 1 | Should Template A be a single chosen open-source project, or should the framework support a plugin system for multiple chatbot UIs? | Maintenance burden |
| 2 | For Template B: should Claude-generated widgets be committed to the repo (static) or generated at build time (dynamic)? | Reproducibility vs freshness |
| 3 | How are widget versions managed when the framework upgrades? Regenerate all widgets? Manual review? | Upgrade safety |
| 4 | Should interactive widgets support undo/redo (state history within the widget)? | UX complexity |
| 5 | Should Template A support WebSocket for bidirectional streaming (user typing indicators, live widget updates)? | Real-time UX |
| 6 | For Template B: should non-React widgets be supported (Vue, Svelte, Web Components)? | Frontend ecosystem |
| 7 | How are widget errors surfaced to the user — inline in the widget, fallback to text, or error toast? | Error UX |
| 8 | Should widgets support A/B testing (multiple widget variants for the same data)? | Product experimentation |
